Fase 1-3: Validação Comercial
Detalhes sobre a entrada do negócio, validação e tratamento de pendências.
WF1 Validação Comercial - Entrada

Este workflow é o ponto de entrada para a validação comercial de novos negócios. Ele garante que cada deal seja direcionado corretamente para a etapa de validação e que um analista seja atribuído.

Gatilho
Deal muda para a etapa "Validação Comercial" em qualquer pipeline.
Condição
Verifica se o deal já passou pela etapa 'Para Validação'.
→
Se Não
Move o deal para a etapa 'Para Validação' no pipeline de Validação.
→
Ação
Atribui um analista (João Vitor / Letícia Ananda) ao deal, seguindo a lógica de rodízio ou pipeline específico.
⚠️
ATENÇÃO · Branch AlugSol duplicado no WF1
O path de Gabriela Bruno está inacessível. Atualmente, todos os deals do tipo AlugSol são direcionados para João Vitor / Letícia Ananda. Isso pode causar sobrecarga ou atrasos se não for corrigido.
WF5 Rede de Segurança - Atribuição Analista

Este workflow atua como uma rede de segurança, garantindo que nenhum deal fique sem um analista atribuído na etapa crítica de validação.

Gatilho
Deal em 'Para Validação' sem analista atribuído por um período definido.
Ação
Atribui um analista (João Vitor / Letícia Ananda) por rodízio para garantir que o deal seja processado.
WF4 Revalidação Comercial

Este workflow é acionado quando um deal precisa ser revalidado comercialmente, geralmente após a correção de pendências.

Gatilho
A propriedade `existe_pendencia_para_validacao_comercial` está vazia E o deal já passou pela etapa 'Para Validação'.
Ação
Move o deal de volta para a etapa 'Para Validação' para uma nova análise.
→
Ação
Atribui um novo analista para garantir uma segunda revisão imparcial.
Fase 4-5: Passagem de Bastão & Alocação
Processos de transição para o Customer Success (CS) e alocação de usinas.
WF Bastão Comercial ↔ CS ID 1653033997

Este workflow gerencia a transição de um deal da equipe comercial para a equipe de Customer Success, direcionando-o para a alocação ou para um fluxo específico de troca de titularidade.

Gatilho
Deal muda para a etapa 'Concluído' (Pipeline Sunne Vendas) ou 'Negócio Fechado' (Pipeline Marketplace/Gestão) ou 'Alocação' (Pipeline iGreen/Alexandria).
Condição
Verifica se a propriedade `troca_de_titularidade` é 'true'.
→
Se Sim
O deal segue um path específico para troca de titularidade.
→
Se Não
Move o deal para a etapa 'Alocação' no Fluxo ACR (Autoconsumo Remoto).
⚠️
ATENÇÃO · Campos de alocação incompletos
Se qualquer um dos 4 campos de alocação (nome_da_usina, gerador_alocacao, id_usina_alocacao_e_rateio, id_gerador_alocacao_e_rateio) estiver vazio, o Workflow de Alocação não avança para a etapa de Cobrança. **Não há alerta explícito no sistema**, o que pode levar a deals travados.

Fluxos n8n
Detalhes dos fluxos n8n integrados ao ciclo operacional.
n8n Criação de Tickets (JSON fornecido anteriormente)

Este fluxo n8n é responsável por agrupar negócios por usina/gerador/alocação e criar tickets de rateio no HubSpot, associando os negócios correspondentes. Ele também atualiza a propriedade ticket_de_cobranca para true nos negócios após a criação do ticket.

Principais Ações:

  • Gatilho: Agendado para rodar em horários específicos (7h, 13h, 19h20) ou acionado manualmente via webhook.
  • Busca de Deals: Realiza uma requisição HTTP (POST) para a API do HubSpot para buscar deals que estão na etapa 1119360529 (provavelmente "Cobrança") e que ainda não possuem um ticket de cobrança associado (ticket_de_cobranca = false). As propriedades hs_object_id, dealname, nome_da_usina, gerador_alocacao, alocacao_ou_substituicao_fluxo_de_alocacao_, consumo_medio_em_kwh, id_usina_alocacao_e_rateio, id_gerador_alocacao_e_rateio são coletadas.
  • Agrupamento de Dados (Code1): Um nó de código JavaScript agrupa os deals encontrados com base em id_usina_alocacao_e_rateio, id_gerador_alocacao_e_rateio e alocacao_ou_substituicao_fluxo_de_alocacao_. Ele calcula o consumo_total para cada grupo e cria uma lista de associações para os deals.
  • Criação de Tickets (HTTP Request6): Para cada grupo de deals, um novo ticket é criado no HubSpot. As propriedades do ticket são preenchidas dinamicamente com informações como subject (assunto), hubspot_owner_id, soma_do_consumo_dos_negocios_associados_kwh, tipo_de_rateio, usina_alocacao_e_rateio, gerador_alocacao_e_rateio, id_usina_alocacao_e_rateio, id_gerador_alocacao_e_rateio e contratos_do_ticket_gerados (definido como false inicialmente). Os deals são associados ao novo ticket.
  • Atualização de Deals (HTTP Request7): Após a criação do ticket, os deals que foram associados a ele têm a propriedade ticket_de_cobranca atualizada para true, evitando que sejam processados novamente.
  • Controle Google Sheets: O fluxo também interage com uma planilha Google Sheets ("Planilha Gamer" - "Controle manual") para registrar o status de "Gerando tickets..." e "Tickets gerados".
n8n Geração de Contratos (Integrado ao WF Ticket 1)

Acionado pelo WF Ticket 1 do HubSpot, este fluxo automatiza a geração dos documentos de contrato para os negócios associados a um ticket de rateio.

Principais Ações:

  • Gatilho: Webhook acionado pelo WF Ticket 1 do HubSpot quando o ticket de rateio muda para a etapa 'Gerar Contratos'.
  • Coleta de Dados: Recebe o ID do ticket e busca todos os deals associados a ele, coletando as propriedades necessárias para a geração dos contratos (ex: amount, dealname, dados do cliente, da usina, etc.).
  • Processamento de Desconto (Desconto por extenso): Um nó de código JavaScript converte o valor numérico do desconto (propriedade desconto) para sua representação por extenso (desconto_extenso), que é utilizada nos contratos.
  • Geração de Documentos: Utiliza um serviço de template (não especificado no JSON, mas inferido) para preencher modelos de contrato (PDF, DOCX) com os dados coletados.
  • Armazenamento Temporário: Os contratos gerados são armazenados temporariamente ou enviados diretamente para o próximo fluxo n8n.
  • Atualização no HubSpot: Atualiza a propriedade contratos_do_ticket_gerados para true no ticket do HubSpot, sinalizando a conclusão desta etapa.
n8n Envio para ClickSign (Integrado ao WF Ticket 2)

Acionado pelo WF Ticket 2 do HubSpot, este fluxo envia os contratos gerados para a plataforma ClickSign para coleta de assinaturas dos envolvidos.

Principais Ações:

  • Gatilho: Webhook acionado pelo WF Ticket 2 do HubSpot quando o ticket de rateio muda para a etapa 'Subir Contratos ClickSign'.
  • Recebimento de Contratos: Pega os contratos gerados pelo fluxo anterior (Geração de Contratos).
  • Criação de Envelope ClickSign: Interage com a API da ClickSign para criar um novo envelope de assinaturas para cada conjunto de contratos.
  • Adição de Signatários e Documentos: Adiciona os signatários (clientes, representantes da Sunne, etc.) e os documentos (termo e regulamento) ao envelope de assinatura.
  • Envio para Assinatura: Envia o envelope para os signatários via ClickSign.
  • Atualização no HubSpot: Atualiza a propriedade contratos_do_ticket_no_clicksign para true no ticket do HubSpot, confirmando o envio.
n8n Checagem de Assinaturas (Integrado ao WF Ticket 3)

Acionado pelo WF Ticket 3 do HubSpot, este fluxo monitora o status das assinaturas dos contratos na ClickSign para garantir que todos os documentos foram devidamente assinados.

Principais Ações:

  • Gatilho: Webhook acionado pelo WF Ticket 3 do HubSpot quando o ticket de rateio muda para a etapa 'Aguardando Assinatura'.
  • Captura Tickets e Deals: Busca tickets no HubSpot que estão na etapa 1242701893 (provavelmente "Aguardando Assinatura") e que já tiveram seus contratos gerados (contratos_do_ticket_gerados = true). Em seguida, captura os deals associados a esses tickets.
  • Verificação de Links ClickSign: Para cada deal associado, verifica as propriedades link_clicksign_termo e link_clicksign_regulamento.
  • Consulta Status ClickSign: Realiza requisições GET para a API da ClickSign, usando os links dos documentos, para checar o status de cada termo e regulamento. Inclui tratamento de erros 404 para identificar documentos não encontrados.
  • Checagem de Conclusão (Code): Um nó de código JavaScript verifica se todos os documentos (termo e regulamento de todos os deals associados) foram encontrados e não retornaram erro 404.
  • Atualização no HubSpot (Move o ticket): Se todos os contratos forem considerados válidos e existentes na ClickSign, a propriedade contratos_de_negocios_do_ticket_assinados é atualizada para True (case-sensitive) no ticket do HubSpot.
n8n Baixar Documentos (Integrado ao WF Ticket 4)

Acionado pelo WF Ticket 4 do HubSpot, este fluxo baixa os documentos assinados da ClickSign e os organiza em pastas específicas no Google Drive.

Principais Ações:

  • Gatilho: Webhook acionado pelo WF Ticket 4 do HubSpot quando o ticket de rateio muda para a etapa 'Baixando Documentos'.
  • Download da ClickSign: Utiliza a API da ClickSign para baixar os documentos finais (contratos assinados) de cada envelope.
  • Organização no Google Drive: Interage com a API do Google Drive para:
    • Criar uma pasta específica para o ticket (se ainda não existir), seguindo uma estrutura padronizada.
    • Fazer o upload dos documentos baixados para essa nova pasta.
  • Atualização no HubSpot: Atualiza as propriedades link_anexos_drive (com o URL da pasta no Google Drive) e documentos_baixados para True (case-sensitive) no ticket do HubSpot.
n8n Cadastro no OPS (Integrado ao WF Ticket 5)

Acionado pelo WF Ticket 5 do HubSpot, este fluxo realiza o cadastro dos negócios associados ao ticket no sistema OPS (Sistema de Operações).

Principais Ações:

  • Gatilho: Webhook acionado pelo WF Ticket 5 do HubSpot quando o ticket de rateio muda para a etapa 'Cadastro no OPS'.
  • Coleta de Dados: Pega os dados necessários do ticket e dos negócios associados, incluindo informações da usina, gerador, cliente e concessionária.
  • Integração com OPS: Envia os dados coletados para a API do sistema OPS para realizar o cadastro dos negócios.
  • Verificação de Sucesso: Monitora a resposta da API do OPS para verificar o sucesso do cadastro.
  • Atualização no HubSpot: Atualiza a propriedade negocios_do_ticket_cadastrados_no_ops para true no ticket do HubSpot.
❌
PROBLEMA CRÍTICO · Concessionária não mapeada no OPS
Se a concessionária do negócio não estiver na lista de 27 mapeadas no workflow n8n, o cadastro no OPS falha silenciosamente.
Solução: Cadastrar manualmente no OPS ou solicitar ao time técnico que adicione a concessionária no mapeamento do workflow n8n.

documentos_baixados Booleano Indica se os documentos foram baixados e organizados no Google Drive. Atualizado pelo n8n (True 🔴 - case-sensitive).

Troubleshooting e Pontos de Atenção
Problemas comuns e como identificá-los no fluxo, com base em observações e dados.
⚠️
ATENÇÃO · Owner do Deal
Se o owner do deal for alterado manualmente após a atribuição inicial pelo workflow, o owner anterior permanece atribuído se não for removido manualmente. Isso pode causar inconsistências na atribuição e notificações.
⚠️
ATENÇÃO · Branch AlugSol duplicado no WF1
O path de Gabriela Bruno está inacessível no WF1. Atualmente, todos os deals do tipo AlugSol são direcionados para João Vitor / Letícia Ananda. Isso pode levar a sobrecarga e atrasos no processo de validação comercial para esses deals.
⚠️
ATENÇÃO · Campos de alocação incompletos
Se qualquer um dos 4 campos de alocação (nome_da_usina, gerador_alocacao, id_usina_alocacao_e_rateio, id_gerador_alocacao_e_rateio) estiver vazio, o Workflow de Alocação não avança para a etapa de Cobrança. **Não há alerta explícito no sistema**, o que pode levar a deals travados sem que a equipe perceba a causa.
⚠️
ATENÇÃO · Ticket sem vínculo com o negócio
Workflows como o WF Ticket 1 (Gerar Contratos) e o WF7 (Rateio Concluído) dependem criticamente da associação entre o ticket e o deal. Sem essa associação, as ações de mover o estágio do deal são ignoradas silenciosamente, resultando em um fluxo que não avança e sem feedback claro do problema.
❌
PROBLEMA CRÍTICO · Concessionária não mapeada no OPS
No fluxo n8n de "Cadastro no OPS" (integrado ao WF Ticket 5), se a concessionária do negócio não estiver na lista de 27 concessionárias mapeadas internamente no workflow, o cadastro no sistema OPS falha silenciosamente. Isso significa que o negócio não será registrado no OPS, mas o HubSpot não receberá um erro explícito.
Solução: É necessário cadastrar manualmente a concessionária no OPS ou solicitar ao time técnico que adicione a concessionária no mapeamento do workflow n8n para que futuras integrações funcionem.
⚠️
ATENÇÃO · WF7 bloqueado por condições não atendidas
O Workflow 7 (Rateio Concluído) exige que 10 propriedades específicas estejam preenchidas e atendam a certas condições para que o deal avance para 'Rateio Concluído'. Se alguma dessas condições estiver faltando ou não for satisfeita, o WF7 é bloqueado e o Deal permanece na etapa 'Em Fluxo de Rateio' indefinidamente, exigindo intervenção manual para identificar e corrigir as propriedades ausentes.
